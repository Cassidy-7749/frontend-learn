# State 與生命週期

State 是 React 元件的「記憶」，它讓元件能夠根據使用者互動、網路回應等事件來更新畫面。理解 state 的運作原理 — 特別是它的非同步更新機制和不可變原則 — 是避免常見 bug 的關鍵。

## useState 基礎用法

`useState` 是最基本的 React Hook，用於在函式元件中新增狀態：

```jsx
import { useState } from 'react';

function Counter() {
  // 宣告一個狀態變數 count，初始值為 0
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>你點擊了 {count} 次</p>
      <button onClick={() => setCount(count + 1)}>
        +1
      </button>
    </div>
  );
}
```

`useState` 回傳一個陣列：
1. **目前的狀態值** — `count`
2. **更新函式** — `setCount`

### 初始值可以是任何型別

```jsx
// 數字
const [count, setCount] = useState(0);

// 字串
const [name, setName] = useState('');

// 布林值
const [isOpen, setIsOpen] = useState(false);

// 物件
const [user, setUser] = useState({ name: '', age: 0 });

// 陣列
const [items, setItems] = useState([]);

// null
const [data, setData] = useState(null);
```

### 惰性初始化

當初始值需要昂貴的計算時，傳入函式可以避免每次渲染都重新計算：

```jsx
// ❌ 每次渲染都會執行 computeExpensiveValue()
const [value, setValue] = useState(computeExpensiveValue());

// ✅ 只在首次渲染時執行
const [value, setValue] = useState(() => computeExpensiveValue());

// 常見場景：從 localStorage 讀取
const [theme, setTheme] = useState(() => {
  return localStorage.getItem('theme') || 'light';
});
```

## 狀態更新是非同步的

這是 React 初學者最常遇到的困惑：呼叫 setState 後，狀態不會立即改變。

```jsx
function Example() {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setCount(count + 1);
    console.log(count); // 仍然是 0！不是 1

    // 因為 setCount 是非同步的
    // count 會在下次渲染時才變成 1
  };

  return <button onClick={handleClick}>Count: {count}</button>;
}
```

### 為什麼是非同步的？

React 刻意將狀態更新設計為非同步，主要出於效能考量：

1. **批次更新** — 將多次 setState 合併為一次重新渲染
2. **一致性** — 確保 props 和 state 在同一次渲染中保持一致
3. **效能最佳化** — 避免不必要的中間渲染

```jsx
function Form() {
  const [name, setName] = useState('');
  const [email, setEmail] = useState('');

  const handleReset = () => {
    // React 會將這兩次更新合併為一次重新渲染
    setName('');
    setEmail('');
    // 而不是渲染兩次
  };
}
```

## 批次更新機制

React 18 引入了**自動批次更新**（Automatic Batching），在任何情境下多次 setState 都會被合併：

```jsx
function BatchingDemo() {
  const [count, setCount] = useState(0);
  const [flag, setFlag] = useState(false);

  console.log('rendered'); // 觀察渲染次數

  const handleClick = () => {
    // React 18：合併為一次渲染
    setCount(c => c + 1);
    setFlag(f => !f);
    // 只會觸發一次 render
  };

  // 即使在 setTimeout 或 Promise 中也會批次更新
  const handleAsync = () => {
    setTimeout(() => {
      setCount(c => c + 1);
      setFlag(f => !f);
      // React 18：仍然只渲染一次
      // React 17：會渲染兩次！
    }, 0);
  };

  // 如果需要強制同步更新，使用 flushSync
  const handleFlush = () => {
    import('react-dom').then(({ flushSync }) => {
      flushSync(() => setCount(c => c + 1));
      // DOM 已更新
      flushSync(() => setFlag(f => !f));
      // DOM 再次更新
      // 共渲染兩次
    });
  };
}
```

<Callout type="info" title="React 17 vs 18 的批次更新差異">
在 React 17 中，只有 React 事件處理函式內的更新會被批次處理。在 `setTimeout`、`Promise`、原生事件監聽器中的更新不會被批次處理。React 18 透過 `createRoot` 啟用了自動批次更新，所有情境下的更新都會自動合併。
</Callout>

## 函式更新器（Functional Updates）

當新的狀態需要基於前一個狀態計算時，應該使用函式更新器：

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  const handleTripleIncrement = () => {
    // ❌ 錯誤：三次都基於同一個 count（0），結果是 1 而非 3
    setCount(count + 1);
    setCount(count + 1);
    setCount(count + 1);

    // ✅ 正確：使用函式更新器，每次基於前一個值
    setCount(prev => prev + 1); // 0 → 1
    setCount(prev => prev + 1); // 1 → 2
    setCount(prev => prev + 1); // 2 → 3
  };
}
```

### 何時使用函式更新器

```jsx
// ✅ 基於前一個值計算 — 使用函式更新器
setCount(prev => prev + 1);
setItems(prev => [...prev, newItem]);
setUser(prev => ({ ...prev, name: 'new name' }));

// ✅ 完全替換為新值 — 直接設定即可
setCount(0);
setName('hello');
setIsOpen(false);
```

## 不可變更新（Immutable Updates）

React 透過**引用比較**來偵測狀態變化。如果你直接修改（mutate）物件或陣列，React 不會偵測到變化，也不會重新渲染。

### 物件的不可變更新

```jsx
function UserProfile() {
  const [user, setUser] = useState({
    name: '小明',
    age: 25,
    address: {
      city: '台北',
      district: '大安'
    }
  });

  // ❌ 直接修改 — React 偵測不到變化
  const handleWrong = () => {
    user.name = '小華';
    setUser(user); // 引用沒變，不會觸發重新渲染
  };

  // ✅ 建立新物件
  const handleCorrect = () => {
    setUser({
      ...user,
      name: '小華'
    });
  };

  // ✅ 巢狀物件也需要展開
  const handleNestedUpdate = () => {
    setUser({
      ...user,
      address: {
        ...user.address,
        city: '高雄'
      }
    });
  };
}
```

### 陣列的不可變更新

```jsx
function TodoList() {
  const [todos, setTodos] = useState([
    { id: 1, text: '學習 React', done: false },
    { id: 2, text: '寫專案', done: false }
  ]);

  // 新增項目
  const addTodo = (text) => {
    setTodos([...todos, { id: Date.now(), text, done: false }]);
  };

  // 刪除項目
  const deleteTodo = (id) => {
    setTodos(todos.filter(todo => todo.id !== id));
  };

  // 更新項目
  const toggleTodo = (id) => {
    setTodos(todos.map(todo =>
      todo.id === id ? { ...todo, done: !todo.done } : todo
    ));
  };

  // 排序（先複製再排序）
  const sortTodos = () => {
    setTodos([...todos].sort((a, b) => a.text.localeCompare(b.text)));
  };

  // 在特定位置插入
  const insertAt = (index, newTodo) => {
    setTodos([
      ...todos.slice(0, index),
      newTodo,
      ...todos.slice(index)
    ]);
  };
}
```

### 陣列操作對照表

| 操作 | 避免使用（直接修改） | 推薦使用（回傳新陣列） |
|------|---------------------|----------------------|
| 新增 | `push`, `unshift` | `[...arr, item]`, `concat` |
| 刪除 | `splice`, `pop` | `filter`, `slice` |
| 取代 | `splice`, `arr[i] = x` | `map` |
| 排序 | `sort`, `reverse` | 先複製 `[...arr].sort()` |

<Callout type="tip" title="使用 Immer 簡化不可變更新">
對於深層巢狀的狀態更新，可以使用 Immer 函式庫。它讓你用「看似直接修改」的語法來撰寫不可變更新：

```jsx
import { useImmer } from 'use-immer';

const [user, updateUser] = useImmer(initialUser);

// 看起來像直接修改，但 Immer 會自動建立新物件
updateUser(draft => {
  draft.address.city = '高雄';
  draft.hobbies.push('游泳');
});
```
</Callout>

## Strict Mode 的雙重渲染

在開發模式下，React 的 `StrictMode` 會刻意**呼叫元件函式兩次**：

```jsx
// main.jsx
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';

createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>
);
```

### 為什麼要雙重渲染？

StrictMode 透過雙重呼叫來幫你找出不純的元件：

```jsx
// ❌ 不純元件 — 雙重渲染會暴露問題
let count = 0;

function Counter() {
  count += 1; // 副作用！
  return <p>呼叫次數：{count}</p>;
  // StrictMode 下會顯示 2 而非 1
}

// ✅ 純元件 — 雙重渲染結果一致
function PureCounter({ value }) {
  return <p>數值：{value}</p>;
}
```

StrictMode 也會對以下項目進行雙重呼叫：
- 元件函式本體
- `useState` 的初始化函式
- `useReducer` 的 reducer 函式
- `useMemo` 和 `useCallback` 的函式

**注意**：這只發生在開發模式，生產環境不會有雙重渲染。

## Class Component 生命週期（簡介）

雖然現代 React 以函式元件為主，但了解 class component 的生命週期有助於理解舊程式碼和 Hook 的設計動機：

```jsx
class MyComponent extends React.Component {
  // 掛載階段
  constructor(props) { }          // 初始化 state
  static getDerivedStateFromProps(props, state) { } // 少用
  render() { }                    // 回傳 JSX
  componentDidMount() { }         // DOM 已建立，適合 API 呼叫

  // 更新階段
  shouldComponentUpdate(nextProps, nextState) { }    // 效能最佳化
  getSnapshotBeforeUpdate(prevProps, prevState) { }  // 少用
  componentDidUpdate(prevProps, prevState) { }       // 更新後的操作

  // 卸載階段
  componentWillUnmount() { }      // 清除計時器、取消訂閱
}
```

### 生命週期 vs Hooks 對照

| Class 生命週期 | 對應的 Hook |
|---------------|------------|
| `constructor` | `useState` 初始值 |
| `componentDidMount` | `useEffect(() => {}, [])` |
| `componentDidUpdate` | `useEffect(() => {}, [deps])` |
| `componentWillUnmount` | `useEffect` 的清除函式 |
| `shouldComponentUpdate` | `React.memo` |
| `this.state` / `this.setState` | `useState` / `useReducer` |

## 狀態提升（Lifting State Up）

當多個元件需要共享同一個狀態時，將狀態「提升」到它們最近的共同父元件：

```jsx
// 需求：兩個溫度輸入框要同步顯示攝氏和華氏

function TemperatureInput({ scale, temperature, onTemperatureChange }) {
  const scaleNames = { c: '攝氏', f: '華氏' };

  return (
    <fieldset>
      <legend>輸入{scaleNames[scale]}溫度：</legend>
      <input
        value={temperature}
        onChange={e => onTemperatureChange(e.target.value)}
      />
    </fieldset>
  );
}

function Calculator() {
  // 狀態提升到共同父元件
  const [temperature, setTemperature] = useState('');
  const [scale, setScale] = useState('c');

  const celsius = scale === 'f'
    ? ((parseFloat(temperature) - 32) * 5 / 9).toFixed(2)
    : temperature;

  const fahrenheit = scale === 'c'
    ? ((parseFloat(temperature) * 9 / 5) + 32).toFixed(2)
    : temperature;

  return (
    <div>
      <TemperatureInput
        scale="c"
        temperature={celsius}
        onTemperatureChange={(t) => {
          setScale('c');
          setTemperature(t);
        }}
      />
      <TemperatureInput
        scale="f"
        temperature={fahrenheit}
        onTemperatureChange={(t) => {
          setScale('f');
          setTemperature(t);
        }}
      />
    </div>
  );
}
```

### 狀態提升的原則

1. 找出需要共享狀態的元件
2. 找到它們最近的共同祖先
3. 將狀態移到該祖先
4. 透過 props 向下傳遞狀態和更新函式

## 多個 State 的管理策略

### 多個 useState vs 一個物件 State

```jsx
// 方案一：多個 useState（推薦用於獨立的狀態）
function Form() {
  const [name, setName] = useState('');
  const [email, setEmail] = useState('');
  const [age, setAge] = useState(0);
}

// 方案二：合併為一個物件（相關聯的狀態）
function Form() {
  const [form, setForm] = useState({
    name: '',
    email: '',
    age: 0
  });

  const updateField = (field, value) => {
    setForm(prev => ({ ...prev, [field]: value }));
  };
}

// 方案三：使用 useReducer（複雜的狀態邏輯）
function Form() {
  const [state, dispatch] = useReducer(formReducer, initialState);
}
```

### 選擇建議

- 2-3 個獨立狀態 → 多個 `useState`
- 多個相關狀態需要一起更新 → 合併為物件
- 複雜的狀態轉換邏輯 → `useReducer`

## 面試重點整理

1. **為什麼 setState 是非同步的？** — 為了批次更新和效能最佳化，React 會將多次 setState 合併為一次重新渲染，避免不必要的中間渲染。

2. **直接設值 vs 函式更新器的差別？** — `setCount(count + 1)` 使用的是閉包中的值，多次呼叫只會基於同一個快照；`setCount(prev => prev + 1)` 每次基於最新值計算，能正確累加。

3. **為什麼不能直接修改 state？** — React 透過引用比較偵測變化，直接修改物件不會改變引用，React 不會觸發重新渲染。且直接修改可能導致不可預期的 bug。

4. **React 18 的自動批次更新是什麼？** — React 18 將批次更新擴展到所有情境（setTimeout、Promise、原生事件等），以前只在 React 事件處理器中才有批次更新。

5. **什麼是狀態提升？** — 當多個子元件需要共享狀態時，將狀態移到它們的共同父元件，透過 props 向下傳遞狀態和更新函式。

6. **StrictMode 雙重渲染的目的？** — 幫助發現不純的元件和意外的副作用，確保元件在多次呼叫下行為一致。僅在開發模式生效。

7. **useState 的惰性初始化是什麼？** — 傳入函式而非值作為初始值，函式只在首次渲染時執行。適用於初始值需要昂貴計算的場景。
