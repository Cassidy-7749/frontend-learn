# 效能優化

React 的宣告式模型讓開發變得直覺，但隨著應用規模成長，效能問題可能逐漸浮現。理解 React 的渲染機制與優化工具，能讓你在需要時精準地提升效能，而不是盲目地到處加上 `memo`。

## Virtual DOM 與 Diffing 演算法

React 效能的基石是 Virtual DOM（虛擬 DOM）。當 state 改變時，React 不會直接操作真實 DOM，而是：

1. **建立新的 Virtual DOM 樹**：根據新的 state 產生新的元素描述
2. **Diffing**：比較新舊 Virtual DOM 樹的差異
3. **Reconciliation**：只將差異部分更新到真實 DOM

```
State 變更 → 產生新 Virtual DOM → Diff 比較 → 最小化 DOM 操作
```

### Diffing 演算法的策略

React 的 Diffing 採用 O(n) 的啟發式演算法，基於兩個假設：

1. **不同類型的元素**會產生不同的樹（直接替換整個子樹）
2. **透過 key 屬性**可以辨識哪些子元素在不同渲染中是穩定的

```jsx
// 類型改變 → 整個子樹重建
// 從 <div> 變成 <section>，所有子元素都會被卸載並重新建立
<div><Counter /></div>
// ↓ 改成
<section><Counter /></section>  // Counter 的 state 會遺失！

// 同類型元素 → 只更新變化的屬性
<div className="old" title="stuff" />
// ↓ 改成
<div className="new" title="stuff" />  // 只更新 className
```

<Callout type="info" title="Fiber 架構">
React 16+ 使用 Fiber 架構重寫了 Reconciliation 引擎。Fiber 讓 React 能夠將渲染工作拆分成小單位，在必要時暫停、恢復或中止渲染。這是 React 18 並行特性（Concurrent Features）的基礎。
</Callout>

## React.memo：避免不必要的重新渲染

當父元件重新渲染時，所有子元件預設也會重新渲染，即使它們的 props 沒有改變。`React.memo` 是一個高階元件，會對 props 進行淺比較（shallow comparison），只在 props 改變時才重新渲染：

```jsx
import { memo, useState } from 'react';

// 沒有 memo：每次父元件渲染，ExpensiveList 都會重新渲染
function ExpensiveList({ items }) {
  console.log('ExpensiveList 渲染');
  return (
    <ul>
      {items.map(item => (
        <li key={item.id}>{item.name}</li>
      ))}
    </ul>
  );
}

// 加上 memo：只在 items 改變時才重新渲染
const MemoizedExpensiveList = memo(ExpensiveList);

function App() {
  const [count, setCount] = useState(0);
  const [items] = useState([
    { id: 1, name: 'React' },
    { id: 2, name: 'Vue' }
  ]);

  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>
        Count: {count}
      </button>
      {/* 點擊按鈕時，MemoizedExpensiveList 不會重新渲染 */}
      <MemoizedExpensiveList items={items} />
    </div>
  );
}
```

### 自訂比較函式

預設的淺比較可能不適合所有場景，你可以提供自訂比較函式：

```jsx
const MemoizedComponent = memo(MyComponent, (prevProps, nextProps) => {
  // 回傳 true 表示 props 相同（不需重新渲染）
  // 回傳 false 表示 props 不同（需要重新渲染）
  return prevProps.id === nextProps.id && prevProps.name === nextProps.name;
});
```

### memo 失效的常見原因

```jsx
function Parent() {
  const [count, setCount] = useState(0);

  // 問題 1：每次渲染都建立新的物件 → memo 失效
  const style = { color: 'red' };

  // 問題 2：每次渲染都建立新的函式 → memo 失效
  const handleClick = () => console.log('clicked');

  // 問題 3：每次渲染都建立新的陣列 → memo 失效
  const data = items.filter(i => i.active);

  return <MemoizedChild style={style} onClick={handleClick} data={data} />;
}
```

這些問題需要搭配 `useMemo` 和 `useCallback` 來解決。

## useMemo 與 useCallback

### useMemo：記憶化運算結果

`useMemo` 會快取一個運算的結果，只在依賴項改變時才重新計算：

```jsx
import { useMemo, useState } from 'react';

function ProductList({ products, filterTerm }) {
  // 只在 products 或 filterTerm 改變時才重新過濾
  const filteredProducts = useMemo(() => {
    console.log('執行過濾運算');
    return products.filter(p =>
      p.name.toLowerCase().includes(filterTerm.toLowerCase())
    );
  }, [products, filterTerm]);

  // 昂貴的統計運算
  const stats = useMemo(() => {
    return {
      total: filteredProducts.length,
      avgPrice: filteredProducts.reduce((sum, p) => sum + p.price, 0) / filteredProducts.length,
      maxPrice: Math.max(...filteredProducts.map(p => p.price))
    };
  }, [filteredProducts]);

  return (
    <div>
      <p>共 {stats.total} 項，平均 ${stats.avgPrice.toFixed(0)}</p>
      {filteredProducts.map(p => <ProductCard key={p.id} product={p} />)}
    </div>
  );
}
```

### useCallback：記憶化函式

`useCallback` 快取一個函式參考，讓它在依賴項不變時保持相同的參考：

```jsx
import { useCallback, useState, memo } from 'react';

const TodoItem = memo(function TodoItem({ todo, onToggle, onDelete }) {
  console.log(`渲染 TodoItem: ${todo.text}`);
  return (
    <li>
      <input type="checkbox" checked={todo.done} onChange={() => onToggle(todo.id)} />
      {todo.text}
      <button onClick={() => onDelete(todo.id)}>刪除</button>
    </li>
  );
});

function TodoList() {
  const [todos, setTodos] = useState([]);

  // 使用 useCallback 確保函式參考穩定
  const handleToggle = useCallback((id) => {
    setTodos(prev => prev.map(t =>
      t.id === id ? { ...t, done: !t.done } : t
    ));
  }, []);

  const handleDelete = useCallback((id) => {
    setTodos(prev => prev.filter(t => t.id !== id));
  }, []);

  return (
    <ul>
      {todos.map(todo => (
        <TodoItem
          key={todo.id}
          todo={todo}
          onToggle={handleToggle}
          onDelete={handleDelete}
        />
      ))}
    </ul>
  );
}
```

<Callout type="warning" title="避免過度使用 useMemo / useCallback">
記憶化本身有成本——需要額外的記憶體來儲存快取值，以及每次渲染時比較依賴項。對於簡單的運算或不傳給 `memo` 子元件的函式，使用 `useMemo` / `useCallback` 反而可能更慢。先測量，再優化。
</Callout>

## key 的效能影響

`key` 不只是消除警告的工具，它直接影響 React 的 Diffing 效率：

### 為什麼不能用 index 當 key

```jsx
// 不好：用 index 當 key
{items.map((item, index) => (
  <TodoItem key={index} {...item} />
))}

// 當在陣列前端插入新項目時：
// 舊：[A(key=0), B(key=1), C(key=2)]
// 新：[D(key=0), A(key=1), B(key=2), C(key=3)]
// React 認為 key=0 的元素從 A 變成 D，key=1 從 B 變成 A...
// → 每個元素都需要更新！

// 好：用唯一 ID 當 key
{items.map(item => (
  <TodoItem key={item.id} {...item} />
))}

// 舊：[A(key=a), B(key=b), C(key=c)]
// 新：[D(key=d), A(key=a), B(key=b), C(key=c)]
// React 知道只需要插入 D，其他元素位移即可
// → 只有一個新元素需要建立！
```

### 用 key 重置元件

改變 `key` 可以強制 React 卸載並重建元件，重置其內部 state：

```jsx
function App() {
  const [userId, setUserId] = useState(1);

  return (
    // 當 userId 改變，Profile 會被完全重建
    <Profile key={userId} userId={userId} />
  );
}
```

## React.lazy 與動態載入

`React.lazy` 讓你延遲載入元件，減少初始載入的 bundle 大小：

```jsx
import { lazy, Suspense } from 'react';

// 動態載入——只在需要時才下載此元件的程式碼
const AdminDashboard = lazy(() => import('./AdminDashboard'));
const UserProfile = lazy(() => import('./UserProfile'));
const Settings = lazy(() => import('./Settings'));

function App() {
  const [page, setPage] = useState('home');

  return (
    <div>
      <nav>
        <button onClick={() => setPage('admin')}>Admin</button>
        <button onClick={() => setPage('profile')}>Profile</button>
        <button onClick={() => setPage('settings')}>Settings</button>
      </nav>

      <Suspense fallback={<div>載入中...</div>}>
        {page === 'admin' && <AdminDashboard />}
        {page === 'profile' && <UserProfile />}
        {page === 'settings' && <Settings />}
      </Suspense>
    </div>
  );
}
```

### 路由級別的程式碼分割

搭配 React Router 使用是最常見的模式：

```jsx
import { lazy, Suspense } from 'react';
import { BrowserRouter, Routes, Route } from 'react-router-dom';

const Home = lazy(() => import('./pages/Home'));
const About = lazy(() => import('./pages/About'));
const Dashboard = lazy(() => import('./pages/Dashboard'));

function App() {
  return (
    <BrowserRouter>
      <Suspense fallback={<LoadingSpinner />}>
        <Routes>
          <Route path="/" element={<Home />} />
          <Route path="/about" element={<About />} />
          <Route path="/dashboard" element={<Dashboard />} />
        </Routes>
      </Suspense>
    </BrowserRouter>
  );
}
```

## Suspense 配合 lazy

`Suspense` 元件讓你宣告式地處理非同步載入的 UI：

```jsx
function App() {
  return (
    <div>
      <header>永遠顯示</header>

      {/* 巢狀 Suspense：不同區域不同的 loading 狀態 */}
      <Suspense fallback={<SidebarSkeleton />}>
        <Sidebar />
      </Suspense>

      <main>
        <Suspense fallback={<ContentSkeleton />}>
          <MainContent />
        </Suspense>
      </main>
    </div>
  );
}
```

<Callout type="tip" title="預載入（Preload）策略">
可以在使用者「可能」需要某個元件之前就開始載入。例如，當滑鼠移到選單項目上時觸發預載入：`const Component = lazy(() => import('./Page')); const preload = () => import('./Page');`。這樣當使用者真正點擊時，模組可能已經下載完成。
</Callout>

## Profiler 工具使用

React DevTools 的 Profiler 能幫你找出效能瓶頸：

### 使用 DevTools Profiler

1. 安裝 React DevTools 瀏覽器擴充
2. 開啟 DevTools → Profiler 分頁
3. 點擊「Record」開始錄製
4. 進行操作
5. 點擊「Stop」查看結果

Profiler 會顯示：
- **Flamegraph**：每個元件的渲染時間
- **Ranked Chart**：按渲染時間排序
- **渲染原因**：為什麼元件會重新渲染

### 程式化 Profiler

React 也提供 `<Profiler>` 元件用於程式化效能測量：

```jsx
import { Profiler } from 'react';

function onRender(id, phase, actualDuration, baseDuration, startTime, commitTime) {
  console.log(`${id} [${phase}]: ${actualDuration.toFixed(2)}ms`);
  // id: Profiler 的 id
  // phase: "mount" 或 "update"
  // actualDuration: 本次渲染的實際時間
  // baseDuration: 沒有任何 memo 時的估計時間
}

function App() {
  return (
    <Profiler id="ProductList" onRender={onRender}>
      <ProductList />
    </Profiler>
  );
}
```

## 列表虛擬化（Windowing）

當需要渲染大量列表（如上千筆資料）時，虛擬化只渲染可見區域的項目：

```
完整列表（10,000 項）：
┌────────────────────┐
│ Item 1             │  ← 不渲染（不可見）
│ Item 2             │
│ ...                │
│ Item 997           │
├────────────────────┤  ← 可見區域開始
│ Item 998           │  ← 渲染
│ Item 999           │  ← 渲染
│ Item 1000          │  ← 渲染
│ Item 1001          │  ← 渲染
│ Item 1002          │  ← 渲染
├────────────────────┤  ← 可見區域結束
│ Item 1003          │
│ ...                │  ← 不渲染（不可見）
│ Item 10000         │
└────────────────────┘
```

### react-window 使用範例

```jsx
import { FixedSizeList as List } from 'react-window';

function VirtualizedList({ items }) {
  const Row = ({ index, style }) => (
    <div style={style} className="list-row">
      {items[index].name} - ${items[index].price}
    </div>

  return (
    <List
      height={400}         // 容器高度
      itemCount={items.length}  // 總項目數

    </List>
  );
}

// 不等高的列表使用 VariableSizeList

  return (
    <VariableSizeList

      width="100%"

```

### @tanstack/react-virtual

更新的替代方案，提供更靈活的 API：

```jsx
import { useVirtualizer } from '@tanstack/react-virtual';

function VirtualList({ items }) {
  const parentRef = useRef(null);

  const virtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 50,
    overscan: 5 // 多渲染前後各 5 項作為緩衝
  });

  return (
    <div ref={parentRef} style={{ height: 400, overflow: 'auto' }}>
      <div style={{ height: virtualizer.getTotalSize(), position: 'relative' }}>
        {virtualizer.getVirtualItems().map(virtualRow => (
          <div
            key={virtualRow.key}
            style={{
              position: 'absolute',
              top: virtualRow.start,
              height: virtualRow.size,
              width: '100%'
            }}
          >
            {items[virtualRow.index].name}
          </div>
        ))}
      </div>
    </div>
  );
}
```

## 避免過早優化

效能優化有其代價——增加程式碼複雜度、可能引入 bug、降低可讀性。遵循以下原則：

### 何時不需要優化

- 元件渲染很快（< 1ms），即使頻繁重新渲染也無感
- 列表項目少（< 100 項），不需要虛擬化
- 函式或物件只在元件內部使用，沒傳給 `memo` 子元件
- 應用整體效能良好，使用者沒有感受到卡頓

### 正確的優化流程

```
1. 先完成功能 → 使用者回報卡頓
2. 用 Profiler 測量 → 找到真正的瓶頸
3. 針對瓶頸優化 → 再次測量確認改善
```

### 常見的不必要優化

```jsx
// 不需要：簡單計算不需要 useMemo
const fullName = useMemo(() => `${firstName} ${lastName}`, [firstName, lastName]);
// 直接寫即可
const fullName = `${firstName} ${lastName}`;

// 不需要：沒傳給 memo 子元件的 callback
const handleClick = useCallback(() => {
  setCount(c => c + 1);
}, []);
// 直接寫即可
const handleClick = () => setCount(c => c + 1);

// 不需要：小陣列的 filter/map
const active = useMemo(() => items.filter(i => i.active), [items]);
// 如果 items 通常只有幾十項，直接寫即可
const active = items.filter(i => i.active);
```

## 效能優化檢查清單

| 問題 | 解決方案 |
|------|---------|
| 子元件不必要的重新渲染 | `React.memo` + `useCallback` / `useMemo` |
| 昂貴的計算 | `useMemo` |
| 初始 bundle 太大 | `React.lazy` + `Suspense` |
| 大量列表 | 虛擬化（react-window / @tanstack/react-virtual） |
| 頻繁的 state 更新 | 批量更新、`useDeferredValue`、`useTransition` |
| 不必要的渲染範圍 | 將 state 下移到需要的元件 |

## 面試重點整理

1. **Virtual DOM**：React 透過比較新舊虛擬 DOM 樹，計算最小差異後批量更新真實 DOM，這是效能的基礎
2. **React.memo**：淺比較 props，避免子元件不必要的重新渲染。搭配 `useCallback` / `useMemo` 確保 props 參考穩定
3. **useMemo vs useCallback**：`useMemo` 記憶運算結果，`useCallback` 記憶函式參考。兩者都不應過度使用
4. **key 的作用**：幫助 Diffing 演算法辨識元素。用唯一 ID 而非 index，尤其在列表會動態增刪時
5. **代碼分割**：`React.lazy` + `Suspense` 實現路由級別或元件級別的動態載入，減少初始 bundle 大小
6. **虛擬化**：大量列表只渲染可見項目。react-window 和 @tanstack/react-virtual 是主流方案
7. **避免過早優化**：先測量，再優化。Profiler 是你最好的朋友
